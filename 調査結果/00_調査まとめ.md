# Digital Billder 請求書 調査まとめ

## 目的

`https://digitalbillder.com/applications` を起点に、ログイン後の画面構造、画面遷移、API 通信、業務フローの全体像を把握する。

## 結論

- このサイトは単純なホームページではなく、ログイン後に使う業務アプリ
- 入口は `/applications`
- 一覧画面から `/invoices/{uuid}` の詳細画面へ入る構造
- 詳細画面は `請求書情報`, `ログ`, `アクション`, `付箋`, `予実管理` の複数タブを持つ
- 中心 API は `https://back.prd.digitalbillder.com/graphql`
- PDF 原本は GraphQL 本文ではなく、署名付き S3 URL から取得している
- 詳細画面はタブごとに都度 API を呼ぶのではなく、初期表示時に多くの情報をまとめて先読みしている可能性が高い

## 画面構造

### 1. 一覧画面

パス:

- `/applications`

役割:

- 担当する請求書の検索と一覧表示

主な検索条件:

- 請求日
- 提出日
- 工事名
- 工事コード
- 工事タグ
- 請求書タグ
- 支店
- 取引先名
- 請求金額（税込）
- 付箋の内容
- 契約有無／請求書種別
- ステータス

主な操作:

- 一括操作
- 連続処理
- アップロード
- ダウンロード
- 電帳法関連
- 並替

実運用上の特徴:

- `要対応` が空でも異常ではない
- 実運用では `対応済み` や `保管済み` の確認が中心になっている可能性が高い
- 実測時は `75件` のデータを確認
- 各行に `確認` リンクがあり、詳細 URL は `/invoices/{uuid}` 形式

### 2. 詳細画面

パス:

- `/invoices/{uuid}`

役割:

- 個別請求書の詳細確認
- PDF 原本確認
- 履歴確認
- 補助業務の確認

上部タブ:

- `請求書情報`
- `ログ`
- `アクション`
- `付箋`
- `予実管理`

#### 請求書情報タブ

確認できたセクション:

- 担当者フロー
- 契約内外管理
- 保存形式
- 工事情報
- 請求書項目
- 請求内訳
- 査定入力
- 差引入力
- 社内用添付資料
- 請求書タグ

確認できた項目例:

- 請求日
- 発行元企業名
- 発行者氏名
- メールアドレス
- 電話番号
- 適格請求書発行事業者登録番号
- 税抜金額
- 消費税額
- 請求金額（税込）
- 取引先コード
- 支払決定金額（税込）

状態表示:

- `保管済みのため編集できません`
- `対応中でないため編集できません`

#### ログタブ

確認できた内容:

- `請求書ログ`
- `日時`, `担当者名`, `アクション`
- `提出`, `編集`, `承認申請`, `承認`, `保管`
- `編集前へ` リンクは `/invoices/logs/{uuid}` 形式

#### アクションタブ

確認できた内容:

- `担当者向けのアクション`
- `請求書保管を取り消す`
- `取消`

注意:

- ボタン確認のみで、実際の操作はしていない

#### 付箋タブ

確認できた内容:

- `付箋作成`
- `確認中のみ表示`

#### 予実管理タブ

確認できた内容:

- `予実管理表`
- `履歴一覧`
- 集計欄
  - 累計発注額
  - 現在請求額
  - 契約外請求金額
  - 経費金額
- 工種ごとの折りたたみ表示

## 通信面の所見

### 一覧画面

- 件数取得と一覧取得は分かれている
- `SearchInvoicesList` は一覧に必要な関連情報も広く取る
- つまり一覧画面も軽い画面ではない

### 詳細画面

- 詳細画面は初回表示で非常に多くの GraphQL を呼ぶ
- `ログ`, `アクション`, `付箋`, `予実管理` を後から開いても追加 GraphQL は観測されなかった
- これらは初期表示時にまとめて先読みされている可能性が高い

## 業務フローの理解

想定される流れ:

1. ログイン
2. `/applications` で請求書一覧を見る
3. ステータスを切り替えて対象を探す
4. `確認` から `/invoices/{uuid}` に入る
5. 請求書情報、ログ、アクション、付箋、予実管理を確認する
6. 必要に応じて外部サービスや管理画面へ進む

## 現時点での重要な理解

- `/applications` は請求書処理の作業台
- `/invoices/{uuid}` は明細確認だけでなく、補助情報を集約した中核画面
- `purchases.digitalbillder.com` は周辺管理機能を受け持つ別サービス
- 画面は SPA 構成で、見た目以上に裏で多くのデータをまとめて取得している
