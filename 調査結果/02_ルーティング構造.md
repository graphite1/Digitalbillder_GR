# ルーティング構造

## 目的

SPA 内の画面遷移構造を整理する。

## 全体像

- `digitalbillder.com/applications` は JavaScript で画面を切り替える業務アプリ
- 1つのアプリ内で URL ごとに画面を切り替える
- 一部画面は別サービス `purchases.digitalbillder.com` に遷移する

## 主なルート分類

### 1. 受領請求書系

- `/applications`
- `/invoices`
- `/invoices/:invoiceId`
- `/invoices/logs/:editLogId`
- `/new/:linkId`
- `/new/in-house/:linkId`

#### `/invoices/:invoiceId` の実画面構造

- タブ切替型の詳細画面
- `請求書情報`, `ログ`, `アクション`, `付箋`, `予実管理` を持つ
- PDF ビューアを画面内に埋め込んでいる

### 2. 発行請求書系

- `/issuer`
- `/issuer/home`
- `/issuer/:invoiceId`
- `/issuer/new`
- `/issuer/formats`
- `/issuer/construction-formats`

### 3. 個人設定・補助系

- `/user-settings`
- `/support`
- `/constructions`
- `/select-box-coordinate`
- `/change-password`
- `/totp-force-setup`

### 4. 外部サービスへ転送される系

- `/notifications`
- `/bulletins`
- `/excel-formats`
- `/option-tables`
- `/departments`
- `/users`
- `/offices`
- `/handling-number`
- `/approval-route`
- `/output-settings`
- `/invoice-settings`

## 初期遷移の考え方

- ログイン状態と権限に応じて遷移先が変わる
- 請求書機能への権限がある場合は `/applications` 方面に入る
- 権限が不足している場合は通知系や外部サービスへ誘導される
- MFA 必須時は `totp-force-setup` へ送られる

## 実運用メモ

- `/applications` ではステータスラジオで表示対象を切り替える
- `要対応`、`対応済み`、`保管済み`、`破棄済` などの状態がある
- 現在の運用状況では `要対応` が空であることがあり、その場合は一覧確認の主対象は `対応済み` または `保管済み` になる
- 一覧行の詳細導線は `/invoices/{uuid}` であることを確認した
